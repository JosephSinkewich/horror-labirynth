# План реализации

---

## 1. Настроить сцену

Стены лабиринта, пол, точки для спавна кристаллов. Сгенерировать навмеш.

- Стартовая точка игрока.

---

## 2. Реализовать игрока

Префаб, коллайдер, компонент для перемещения.

- Отдельный компонент камеры: FPS, вращение мышью.
- Shift = бег.
- Блокировка управления при победе/поражении.

---

## 3. Реализовать мутанта1

Префаб, коллайдер, навмеш-агент, ии решающий патрулировать или преследовать. Глаза не нужны.

---

## 4. Реализовать мутанта2

Префаб, коллайдер, навмеш-агент с другими параметрами, отличающимися от первого мутанта, другой ии. Без глаз.

---

## 5. Реализовать кристалл

Префаб, коллайдер, игроку прописать компонент для сбора кристаллов.

- Визуальное выделение: вращение или простая анимация/свечение.

---

## 6. Реализовать чистый класс подсчёта кристаллов

Который будет биндиться зенжектом. Количество собираемых кристаллов будет настраиваться в биндящемся SO настроек баланса.

- События `Collected`, `AllCollected` — для UI, выхода и спавнера.

---

## 7. Реализовать HUD

С количеством собранных кристаллов. Он прослушивает события чистого класса подсчёта кристаллов.

---

## 8. Реализовать систему спавна кристаллов

На уровне в ключевых точках. Переставать спавнить кристаллы, когда их уже достаточно на уровне для прохождения.

- При старте уровня — заспавнить нужное количество (из Balance SO, ориентир 5–10).
- *(В ТЗ — «рандомные точки»; ключевые точки на сцене — допустимый упрощённый вариант.)*

---

## 9. Реализовать логику поражения

При столкновении с монстром. Показывается окно поражения с кнопкой рестарт.

- Блокировка управления.
- Restart → перезагрузка сцены.

---

## 10. Реализовать выход из лабиринта

Выход переходит в режим «открыт», когда чистый класс подсчёта кристаллов сообщает, что все кристаллы собраны. При попадании игрока в выход показывается окно победы с количеством собранных кристаллов и кнопкой рестарт.

## Что было упущено в исходном списке (по ТЗ)


| Тема                                       | Почему важно                                             |
| ------------------------------------------ | -------------------------------------------------------- |
| **Отдельный класс камеры**                 | Явное требование ТЗ; не смешивать с движением            |
| **Shift = бег**                            | Отдельное требование к управлению                        |
| **CharacterController + блокировка стен**  | Критерий приёмки                                         |
| **Точки патруля / маршруты врагов**        | ТЗ про сцену и поведение врагов                          |
| **Радиус обнаружения + возврат к патрулю** | Минимальный ИИ по ТЗ (без raycast — ок)                  |
| **GameSession / состояния игры**           | Победа, поражение, рестарт — отдельный класс-координатор |
| **Блокировка ввода при win/lose**          | Иначе игрок продолжит бегать под UI                      |
| **Restart = перезагрузка сцены**           | Кнопка есть в UI, но нужна реализация                    |
| **Визуал кристалла** (вращение)            | Желательно по ТЗ                                         |
| **Стартовая точка игрока на сцене**        | Явно в ТЗ про лабиринт                                   |
| **Zenject / installer / SO**               | Твоя архитектура — добавить шаг настройки                |
| **Build Settings, Play без ошибок**        | Техническое требование                                   |
| **README: управление + архитектура**       | Обязательно для сдачи работы                             |


### Не критично, но полезно

- Закрытый выход должен **не давать победу**, пока не собраны все кристаллы (логика + опционально визуал «закрыто»).
- Слои физики заранее — меньше сюрпризов с коллизиями враг/игрок/стены.
- Два мутанта с **разным** поведением — не только разные цифры в agent, но и ощутимая разница (скорость vs радиус, разные патрули).
